# 命令与交互设计

这是一份开发文档，用于给维护者整理 `goani-cli` 当前命令体系、交互问题和后续演进方向。

它不是面向最终用户的使用说明，也不替代 [usage.md](../usage.md) 这类用户文档。

这份文档不是简单罗列命令，而是整理 `goani-cli` 当前已经具备的能力，并基于这些能力重新设计一套更顺手的交互模型。

目标有两个：

1. 让当前 CLI 更容易理解、帮助信息更清晰。
2. 为后续统一 TUI 留出稳定的命令结构，不让功能越长越乱。

---

## 设计目标

`goani-cli` 现在已经不只是“搜一下然后播一下”的小工具了。它已经有：

- 搜索
- 播放
- 播放器配置
- 片源订阅管理
- 更新
- 版本信息
- 实时搜索 TUI

如果继续只靠“每次都输入一串 `goani ...` 子命令”来承载所有功能，后面加上历史播放、继续观看、上一集、下一集、清晰度切换之后，用户体验会越来越碎。

因此更合理的方向是：

- 保留 CLI，方便脚本、快捷命令、文档示例和非交互环境。
- 把人类日常使用的主流程逐步收敛到 TUI。

一句话总结就是：

`CLI for scripting, TUI for humans.`

---

## 当前功能现状

当前实际可用的命令可以整理成这 4 类：

### 1. 内容消费

- `goani search <keyword>`
- `goani search --interactive <keyword>`
- `goani play <keyword>`

### 2. 配置管理

- `goani config player <name> <path>`
- `goani config player default <name>`

### 3. 片源管理

- `goani source list`
- `goani source sub <url> [name]`
- `goani source unsub <url>`
- `goani source refresh`
- `goani source reset`

### 4. 程序信息

- `goani update`
- `goani version`

这个结构本身没有错，但交互上已经暴露出一些问题。

---

## 当前交互问题

### 1. 人类主路径过于依赖一次性命令

目前大多数使用方式都像这样：

```bash
goani search 葬送的芙莉莲
goani play 葬送的芙莉莲
goani source list
goani config player mpv "/path/to/mpv"
```

这对熟悉命令行的人没问题，但对日常使用来说，操作负担还是偏大。

用户真正想做的通常不是“执行命令”，而是：

- 找一部番
- 继续看上次那一集
- 换播放器
- 刷新片源
- 看看默认源是不是正常

这些动作都更像“界面操作”，不是“记忆命令”。

### 2. 当前命令粒度偏技术化

例如：

- `source sub`
- `source unsub`
- `source refresh`
- `config player default`

这些命令对于开发者很清楚，但对普通用户来说，它们更像“内部术语”，不是“用户意图”。

### 3. 主流程被拆碎

当前“搜索 -> 选源 -> 选番 -> 选集 -> 播放”基本已经是一个流程，但入口分散在：

- 普通搜索
- 实时搜索
- 直接播放

这会导致以后功能越多，入口越多，用户越难判断“应该从哪个命令开始”。

### 4. 帮助信息目前仍偏命令参考，不够像操作指南

`-h` 输出应该让用户立刻知道：

- 这个命令适合什么时候用
- 直接改 `config.json` 行不行
- 我是普通用户的话更推荐点哪里、走哪条路径

也就是说，帮助信息应该是“操作指引”，不只是“语法说明”。

---

## 建议的总体方向

建议把 `goani` 设计成“双入口模式”。

### 模式 A：TUI 主入口

适合人类日常使用。

```bash
goani tui
```

在交互式终端里，直接进入主界面。

主界面建议承载：

- 搜索
- 最近播放 / 历史播放
- 继续观看
- 选择片源
- 刷新片源
- 配置播放器
- 设置默认播放器
- 检查更新
- 查看版本

### 模式 B：CLI 显式命令

适合脚本、文档、快捷调用和非交互环境。

```bash
goani play 葬送的芙莉莲
goani source refresh
goani config player mpv "/usr/bin/mpv"
```

CLI 不应该消失，因为它有几个天然优势：

- 可脚本化
- 可复制粘贴
- 易写文档
- 适合 SSH / CI / 非交互环境

所以正确方向不是“砍掉 CLI”，而是“把 TUI 变成默认的人类入口”。

## 推荐的命令模型

基于当前功能，我建议命令分为 3 层。

### 第一层：面向普通用户的入口

- `goani`
  - 显示帮助、命令说明和推荐路径
- `goani tui`
  - 进入主 TUI
- `goani search [keyword]`
  - 快速搜索
- `goani play <keyword>`
  - 快速搜索并播放

这层命令应该尽量少、尽量直观。

### 第二层：面向配置与维护的命令

- `goani config ...`
- `goani source ...`
- `goani update`
- `goani version`

这层命令保留子命令结构没有问题，因为本来就是低频操作。

### 第三层：内部实现层

这层不直接暴露给用户，而是体现在代码组织中，例如：

- 搜索服务
- 播放服务
- 历史记录服务
- 配置服务
- 片源服务
- TUI 页面状态机

用户层的命令应该尽量薄，不要把复杂逻辑都堆进命令处理函数。

---

## 推荐的 TUI 主流程

如果后面要逐步把整个应用统一成 TUI，建议主界面先只做最核心的 5 个入口。

### 首页

- 继续观看
- 搜索动漫
- 最近播放
- 媒体源管理
- 设置

### 搜索页

搜索页应该成为后续最重要的一页。

建议支持：

- 输入关键词后实时搜索
- 展示片源状态和结果数
- 上下移动选择
- 回车查看番剧列表
- `Tab` 或左右键切换源 / 番剧 / 剧集焦点
- `Esc` 返回上一层

### 详情页 / 剧集页

选中番剧后进入剧集页。

建议支持：

- 剧集列表
- 默认选中最近播放集数
- 回车播放
- 显示当前使用的片源
- 显示可切换清晰度
- 支持“下一集 / 上一集”快捷操作

### 播放后控制

后面如果要做历史播放和连续观看，这一层会非常关键。

建议逐步加：

- 标记已播放进度
- 下一集
- 上一集
- 重播本集
- 切换播放器
- 切换片源
- 切换清晰度

### 设置页

建议把所有维护型操作收进设置页，而不是让用户记命令。

可以包含：

- 默认播放器
- 已配置播放器路径
- 当前订阅列表
- 添加 / 删除订阅
- 刷新片源
- 重置默认源
- 查看配置文件位置
- 打开配置文件

## CLI 与 TUI 的职责边界

为了避免以后命令和界面相互打架，建议尽早定下边界。

### TUI 负责

- 高频使用
- 需要反复选择的流程
- 多步骤交互
- 适合视觉呈现的状态信息

典型例子：

- 搜索
- 选番
- 选集
- 历史播放
- 清晰度切换
- 连播

### CLI 负责

- 一次性动作
- 可脚本化动作
- 配置写入
- 自动化调用
- 调试 / 维护

典型例子：

- `goani source refresh`
- `goani source sub <url>`
- `goani config player mpv "..."`
- `goani version`
- `goani update`

## 帮助信息设计建议

后续所有帮助信息建议遵循同一套结构：

### 1. 先说用途

不是一上来就扔“用法”，而是先告诉用户“这个命令是干嘛的”。

### 2. 再说语法

```text
用法:
  goani config player <name> <path>
```

### 3. 再给高频示例

优先展示用户最常复制的例子。

### 4. 再补充相关配置说明

例如：

- 配置文件位置
- 是否也能直接编辑 JSON
- 当前命令更适合命令行还是更适合在 TUI 里完成

帮助信息应该是用户的“即时操作说明书”。

---

## 配置交互建议

当前配置已经统一收敛到 `config.json`，这是很好的方向。

下一步建议明确支持两种配置方式：

### 方式 A：命令式配置

适合快速设置。

```bash
goani config player mpv "/path/to/mpv"
goani config player default mpv
```

### 方式 B：文件式配置

适合高级用户直接编辑。

```json
{
  "player": {
    "default": "mpv",
    "paths": {
      "mpv": "/usr/bin/mpv"
    }
  },
  "sources": {
    "subscriptions": [
      {
        "url": "https://sub.creamycake.org/v1/css1.json",
        "name": "默认源"
      }
    ]
  }
}
```

建议后续在 TUI 设置页里也围绕这份统一配置来做，不要再分裂出新的散乱配置入口。

## 推荐的演进路径

为了不一下子改太猛，建议按下面顺序演进。

### 第一阶段：先把文档和帮助信息理顺

- 顶层帮助更清晰
- `config -h` / `source -h` 更像操作指南
- 命令文档和使用文档一致

### 第二阶段：把搜索做成稳定的 TUI 页面

- 搜索输入
- 结果列表
- 源选择
- 番剧选择
- 剧集选择

### 第三阶段：补一个真正的首页

```bash
goani tui
```

直接进入首页，而不是只打印帮助。

### 第四阶段：把高频功能收口到 TUI

- 历史播放
- 继续观看
- 下一集 / 上一集
- 清晰度选择
- 设置页

### 第五阶段：保留 CLI 作为自动化接口

CLI 继续存在，但不再承担所有“人类日常交互”的职责。

---

## 推荐结论

基于当前功能，我的建议非常明确：

1. `goani` 未来应该默认进入 TUI 主界面。
2. `goani tui` 作为人类主入口更合适，`goani` 本身保留为帮助页。
3. `search`、`play`、`config`、`source` 继续保留，作为显式 CLI 接口。
4. 高频流程进 TUI，低频维护留 CLI。
5. 所有配置继续统一写入 `config.json`。
6. 后续新增功能优先考虑“它应该出现在 TUI 的哪一页”，而不是先想“要不要再加一个子命令”。

这套结构的好处是：

- 对用户更友好
- 对文档更清晰
- 对后续功能扩展更稳定
- 不会把项目变成一堆越来越难记的子命令

---

## 当前建议的用户心智模型

最后可以把 `goani` 简化成一句话：

- 想正常使用，就输入 `goani`
- 想进入交互模式，就输入 `goani tui`
- 想快速直达，就输入 `goani search` 或 `goani play`
- 想维护配置，就输入 `goani config` 或 `goani source`

如果后续所有设计都围绕这个心智模型展开，整个项目会清爽很多。

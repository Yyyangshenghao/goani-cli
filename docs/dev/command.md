# 命令与交互设计

这是一份维护者文档，用来说明当前已经实现的命令模型，以及后续值得继续演进的方向。

它不是用户说明书，也不替代 [usage.md](../usage.md)。

---

## 当前已实现的命令模型

现在的命令结构可以分成三层。

### 1. 顶层入口

- `goani`
  - 默认显示帮助
- `goani tui`
  - 进入主 TUI
- `goani search <keyword>`
  - 经典搜索入口
- `goani play <keyword>`
  - 搜索后直接进入播放流程

### 2. 维护命令

- `goani config ...`
- `goani source ...`
- `goani update`
- `goani version`

### 3. 内部命令

- `proxy-hls`

这个命令只给播放器兼容层使用，不应该出现在普通帮助里。

## 现在的 TUI 流程

当前 `goani tui` 已经不是一个空壳入口，而是一个完整的交互主线。

入口和页面关系更准确地说是：

1. 首页
2. 从首页进入搜索页
3. 在搜索页查看聚合后的番剧结果
4. 进入聚合番剧列表页
5. 进入聚合选集页
6. 进入线路选择页
7. 播放成功后进入播放状态页
8. 配置页、媒体源页、版本信息页都从首页单独进入

关键体验点：

- 搜索页会尽量把多个片源的同名番剧先聚合。
- 番剧列表支持本地二次过滤。
- 选集页会把同一集的多片源候选线路合并展示。
- 选集页支持顺序和倒序切换。
- 选集页支持直接输入集数。
- 线路解析有加载页，最多等待 5 秒。
- 线路页会标明线路所属片源。
- `PotPlayer + m3u8` 会走本地 HLS 代理。
- 播放状态页已经支持上一集、下一集、返回选集、返回番剧列表和返回首页。
- 出错后尽量留在 TUI 流里，而不是直接退回桌面。

## 当前设计原则

### 1. `goani` 负责说明，`goani tui` 负责交互

这两个入口的职责应该尽量稳定，不要互相抢。

- `goani` 用来告诉用户“有哪些命令、怎么开始”
- `goani tui` 用来承载人类日常操作

### 2. CLI 继续保留

CLI 不是旧东西，它是脚本、自动化和调试的稳定接口。

因此更合适的策略是：

- 高频人机交互交给 TUI
- 低频维护和自动化保留 CLI

### 3. 配置优先收口到 `config.json`

命令式配置和文件式配置都可以保留，但底层都应该落到同一份配置文件里。

现在比较明确的规则是：

- `goani config player <name> <path>` 只保存路径
- `goani config player default <name>` 只设置默认播放器
- 片源订阅和播放器配置都保存在 `config.json`

## 帮助信息应该怎么写

后续命令帮助建议统一成这四段：

1. 先说用途
2. 再说语法
3. 再给高频示例
4. 最后补充与配置或 TUI 的关系

帮助信息要像“快速操作指南”，不要只像参数清单。

---

## 后续可以继续做什么

如果以后继续扩展，我建议优先考虑这些方向：

- 历史播放
- 继续观看
- 清晰度切换
- 片源和播放器的设置页细化

这些都更适合放进 TUI，而不是继续增加一堆新的命令层级。

---

## 不建议的方向

- 不要把 `goani tui` 再拆成很多 `goani tui xxx` 的嵌套命令。
- 不要把所有高频交互都留在 CLI。
- 不要让文档同时讲一套“理论模型”和一套“实际行为”。

如果后续要改命令模型，建议先改文档，再改帮助输出，最后再改实现，避免用户看到三套说法。
